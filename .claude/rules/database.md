# Database Rules — SQLAlchemy Async + PostgreSQL/SQLite

## Core Discipline

- **All DB access through repositories**: services never import `Session` or `AsyncSession` directly; they call repository methods.
- **Explicit transactions**: always use `async with session.begin()` or `async with session` + `await session.commit()`. Never rely on auto-commit.
- **No raw SQL strings** for business logic: use `sqlalchemy.select()`, `update()`, `insert()` expressions. `text()` is allowed only for DDL utilities or one-off diagnostics.

## Models

- Inherit from `Base = DeclarativeBase()` — SQLAlchemy 2.x style.
- Use `mapped_column()` and `Mapped[T]` type annotations — not `Column()`.
- Table names: `snake_case`, plural (e.g. `telemetry_events`, `zone_counts`).
- Column names: `snake_case`.
- Primary keys: prefer `Integer` auto-increment for high-volume tables (telemetry_events); UUID for entities referenced externally.
- Always set `nullable=False` explicitly — don't rely on defaults.
- `TIMESTAMPTZ` for all timestamps; store UTC, display in client timezone.

## Concurrency-Critical Patterns

### Zone Entry Counter (atomic increment)
```python
# CORRECT — single SQL statement, no ORM read-modify-write
await session.execute(
    update(ZoneCount)
    .where(ZoneCount.zone_id == zone_id)
    .values(entry_count=ZoneCount.entry_count + 1)
)

# WRONG — read then write races under concurrency
zone = await session.get(ZoneCount, zone_id)
zone.entry_count += 1  # race condition
```

### Fault → Mission Cancellation (serializable transaction)
```python
# Use SELECT FOR UPDATE to lock vehicle row, then act atomically
vehicle = await session.execute(
    select(VehicleState)
    .where(VehicleState.vehicle_id == vehicle_id)
    .with_for_update()
)
# cancel mission + create maintenance record in same transaction
```

### Fleet Aggregate State (never in-process counters)
```python
# CORRECT — DB does the counting
result = await session.execute(
    select(VehicleState.status, func.count().label("count"))
    .group_by(VehicleState.status)
)

# WRONG — maintains counters in Python dicts, breaks under multi-worker
```

### VehicleState Upsert
```python
# PostgreSQL
await session.execute(
    insert(VehicleState)
    .values(...)
    .on_conflict_do_update(
        index_elements=["vehicle_id"],
        set_={...}
    )
)
```

## Session Lifecycle

- Create `AsyncSessionLocal` factory with `autocommit=False`, `autoflush=False`, `expire_on_commit=False`.
- Use FastAPI dependency injection: `async def get_session() -> AsyncGenerator[AsyncSession, None]`.
- Never keep sessions alive across requests.
- For tests: use a separate test engine; roll back after each test (not truncate).

## Migrations (Alembic)

- Every schema change = a new migration: `alembic revision --autogenerate -m "<description>"`.
- Review autogenerated migrations before applying — Alembic misses some patterns (e.g. enum changes).
- `alembic upgrade head` in startup script or lifespan event (dev only; CI applies separately).
- Never drop columns or tables in the same migration that adds new ones unless clearly intentional.

## Performance

- Index `(vehicle_id, timestamp)` on `telemetry_events` — primary query pattern for anomaly lookups.
- Index `(vehicle_id)` on `anomalies` + `detected_at` for range filters.
- Avoid `SELECT *` — always select specific columns in read paths.
- Limit result sets: default pagination of 100 for telemetry queries; never unbounded.

## SQLite Dev Mode

- Use `aiosqlite` driver: `sqlite+aiosqlite:///./fleet.db` or `sqlite+aiosqlite:///:memory:` for tests.
- SQLite does not support `SKIP LOCKED` or `SELECT FOR UPDATE` — integration tests for concurrency **must** use PostgreSQL (via Docker or CI service).
- Document this limitation in README.
